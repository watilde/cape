## セッション終了時チェックリスト

### 🔴 必須（スキップ不可）

- [ ] **アーキテクチャ・実装方針の変更がある場合**：Option A/B/C 評価を記述し、DA・CM・POAに同時通知した
- [ ] **DAスペックに仕様未確定箇所・言語的表現のみのアニメーション記述があった場合**：DAに差し戻した（独断実装していない）
- [ ] **実装リスクを認識した箇所がある場合**：DAのImplementation Notesセクションを確認し、記述がなければDAに要求した
- [ ] **Open Questionsが存在する場合**：全OQにステータス（OPEN / CLOSED / 次セッション前完了義務）を記録した
- [ ] **今セッションで追加・更新されたDoD個人項目**：このファイルに反映した

### 🟡 推奨（理由付きでスキップ可）

- [ ] アーキテクチャドキュメント（1_architecture.md）との整合性を確認した
- [ ] テスト戦略（追加・変更した機能のユニットテスト方針）を記述した
- [ ] DAとの視覚的変更のペアレビューが必要な変更がある場合、依頼を送った

## DoD個人項目（恒久ルール）

### アーキテクチャ変更ルール（task-1779178940125 / POA K-02より格上げ）

> アーキテクチャ・実装方針の変更・選択が生じた場合は必ずOption A/B/C形式で評価を記述し、最終選択根拠をDA・CM・POAに通知する。

**適用トリガー**:
- 新規モジュール・レイヤーの追加
- 既存の状態管理方式の変更（例：Context → Redux）
- CSSアーキテクチャの変更（例：CSS変数 → JS計算）
- パフォーマンス最適化方針の変更
- データ永続化方式の変更

**Option評価フォーマット**:
```
Option A: [実装方法]
  メリット: [記述]
  デメリット: [記述]
  リスク: [記述]
Option B: ...
Option C: ...
推奨: Option X — 理由: [DA・CM・POAが判断できる形で記述]
```

### DAスペック受け取り確認ルール（DA DT-02連動 / task-1779179987205より）

> DAスペックを受け取った際に以下を必ず確認し、問題がある場合は実装開始前にDAに差し戻す。

**チェック項目**:
1. アニメーション仕様に言語的表現（「ゆっくり」「なめらかに」等）のみの記述がないか → あればDAに差し戻す
2. 仕様未確定箇所（「DevAに委ねる」等の表現）がないか → あればDAに差し戻す（独断実装禁止）
3. 実装リスクが認識される箇所にImplementation Notesが記述されているか → なければDAに追記要求

**差し戻しフォーマット**:
```
[差し戻し] DA → DevA スペック確認
対象: [コンポーネント名 / セクション名]
問題: [言語的表現のみ / 仕様未確定 / Implementation Notes欠如]
必要な情報: [具体的に何が必要か]
期限: [次ターン / 次セッション開始前]
```

### 変更通知ルール（task-1779179987205 / POA T-06より再起動）

> 以下の変更を行う場合、差分プランではなくメインアウトプットに記録し、DA・CM・POAに同時通知する。

**対象変更の定義**:
- CSSアーキテクチャの変更
- 状態管理パターンの変更
- データ構造（型定義）の変更
- サードパーティライブラリの追加・変更・削除
- パフォーマンス計測結果に基づく仕様変更

**通知方法**: アウトプットJSONの `output` フィールド内に `architecture_changes` セクションを追加し、変更内容・理由・影響範囲を記述する。

### Open Questions管理ルール（VP-02 / POA P-04より）

> Open Questionsは以下のステータスのいずれかを必ず付与する。

| ステータス | 意味 | アクション |
|---|---|---|
| `OPEN` | 調査・計測が必要、セッション内未完了 | 次セッション開始前完了義務として記録 |
| `CLOSED` | 回答済み・完了済み | 完了根拠（数値・証跡）を記録 |
| `BLOCKED` | 外部依存で進行不可 | ブロッカーと担当者を明記 |
| `DEFERRED` | 意図的に次フェーズへ先送り | 先送り理由とフェーズを明記 |

**ゾンビ化防止ルール**: `OPEN` ステータスのOQは2セッションを超えてそのままにしない。2セッション目終了時点で未完了の場合、CMが強制的に次セッションの最初のアジェンダとして設定する。
