## KPT

### Keep

- POAがペルソナ（Yuki・Hana・Kenji）とMVVを既存資産として活用し、「かわいいエフェクト」を3トリガー×感情ゴール構造で即座に定義できた——既存コンテキストをアンカーにする手法は再現性が高い
- DAのハンドオフノートが具体的かつ技術的に正確で（取り消し線の::after疑似要素、max-height問題、prefers-reduced-motion必須化）、DevAの設計判断の迷いをゼロにした——次回も同水準のハンドオフノートを維持する
- パーティクルエフェクトをLottie・Framer Motion不使用のVanilla CSS + Web Animations APIで実装したことで、外部依存を最小化しながらGPUアクセラレーションを確保した技術選定
- POAの受け入れ基準（AC-001-1〜AC-004-2）がすべてコンポーネントに1対1でマッピングされており、実装の追跡可能性が高かった
- prefers-reduced-motion対応・aria-hidden付与・44×44pxタッチターゲット確保など、アクセシビリティ要件をDAとDevAが両側から担保した

### Problem

- タスク編集機能（テキスト変更）をv1.0.0スコープ外とする判断が、POA成果物の中で明示的に説明されていなかった——DAとDevAが『意図的除外か見落としか』を判断できず、曖昧さが残った
- ブリーフで指定されたカラー（Violet #B197FC・Navy #1C2541）とブランドガイドライン（Sweet Pink系）の衝突をDAが独断で統合処理した——チームで設計判断を共有するステップが抜けており、後から覆すコストが高い変更が無言で確定した
- ElectronのcontextIsolation・セキュリティポリシーに関する制約がブリーフに記載されておらず、DevAがLocalStorage直接アクセスで実装した——本番Electron環境では動作しない可能性があり、今の段階でリスクが未解決のまま
- DoRが未達のままセッションをGOした——結果的に既存ペルソナで救われたが、DoRチェックをバイパスする習慣化はプロセスの形骸化につながる

### Try

- POAは『意図的にスコープ外にした機能』を成果物の中に明示的なセクション（例：Explicitly Excluded Features）として記載する——DAとDevAが仕様の穴と意図的除外を区別できるようにする
- DAがブリーフ指定要素（カラー・フォントなど）とブランドガイドラインに衝突が生じた場合、独断で処理せずCMに一報してチーム確認ステップを挟む——簡易スラック報告でも可、透明性を優先
- DoRチェックリストに『Electron固有の技術制約の事前確認（contextIsolation・ipcRenderer利用有無・preload設定）』を追加し、DevAがDoR段階で技術リスクをボトムアップで申告できる仕組みを作る
- DoR未達でGOする場合は、CMが明示的に『条件付きGO（既存資産〇〇で代替）』として記録し、次フェーズのハンドオフに懸念事項として引き継ぐ——暗黙のGOではなくトレーサブルな判断にする
